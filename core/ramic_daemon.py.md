这段 Python 代码是 **RAMIC Bridge** 的后端守护进程（Daemon）。它扮演着一个“中转站”的角色：**一头连着网络 TCP 端口，另一头连着 Cadence Virtuoso 的标准输入输出（IPC）**。



通过这个脚本，外部程序（如你的 Web 浏览器、AI 代理或本地脚本）可以通过网络向 Virtuoso 发送 SKILL 指令，并同步获取执行结果。

---

### 1. 核心工作流程
1.  **启动**：Virtuoso 通过 `ipcBeginProcess` 运行这个脚本。
2.  **监听**：脚本在指定端口（如 65432）开启 TCP 监听。
3.  **接收**：外部客户端发送一段 JSON 格式的数据（包含 SKILL 代码）。
4.  **过滤与预处理**：脚本检查代码安全性，并将其格式化为单行。
5.  **透传**：通过 `sys.stdout` 将代码传给 Virtuoso。
6.  **读取**：通过 `sys.stdin` 等待 Virtuoso 的反馈（使用特殊控制字符识别边界）。
7.  **回传**：将 Virtuoso 的执行结果通过 TCP 套接字返回给客户端。

---

### 2. 关键功能模块详解

#### **A. 非阻塞标准输入 (Non-blocking stdin)**
```python
fcntl.fcntl(sys.stdin.fileno(), fcntl.F_SETFL, ...)
```
* **作用**：默认情况下，读取 stdin 会导致程序阻塞（停下来等）。为了保证网络响应的灵敏度，脚本将 stdin 设为非阻塞模式。配合 `time.sleep(0.001)`，脚本可以在没有数据时短暂释放 CPU，防止死循环。

#### **B. 结果读取算法 (`read_result`)**
Virtuoso 的 SKILL 脚本中使用了 `sprintf(nil "%c%L%c" 2 result 30)`。
* **STX (\x02)**：正文开始。
* **NAK (\x15)**：错误发生。
* **RS (\x1e)**：记录分隔符（结束标记）。
脚本通过循环不断读取字符，直到撞见 `RS`，从而完整地提取出一次 SKILL 执行的结果，避免读到不完整的数据或下一次请求的数据。

#### **C. 安全防线 (`_check_skill`)**
```python
_BLOCKED_FNS = re.compile(r'(?<!["\w])(shell|system|ipcBeginProcess|...)\s*\(')
```
* **作用**：这是一个简单的“沙箱”机制。它禁止通过这个桥接器执行危险的系统级 SKILL 函数（如 `shell` 或 `system`）。
* **原理**：使用正则表达式匹配代码。为了防止误判，它先用 `re.sub` 剔除掉字符串常量（比如代码里正好有个字符串叫 "system" 是允许的，但调用 `system()` 就会被拦截）。

#### **D. SKILL 代码清洗 (Handling & Formatting)**
```python
skill = re.sub(r'"[^"]*"|;[^\n]*', ...) # 移除注释
skill = ' '.join(skill.split())          # 压缩为空格
```
* **为什么要这么做？** Virtuoso 的 `ipcBeginProcess` 通常是**按行**触发数据处理器的。如果发送多行代码，可能会触发多次回调导致逻辑错乱。
* **逻辑**：脚本先移除 SKILL 的分号注释（`;`），然后将多行代码合并成一行，确保 Virtuoso 的 `evalstring` 能一次性完整执行。

### 4. 为什么这个脚本比直接写 SKILL 更高级？
通常的 SKILL IPC 只能做简单的单向触发。这个 Python 脚本增加了一个**网络抽象层**：
* 它解决了 **跨语言通信**：让不懂 SKILL 的 Python 开发者也能控制 Cadence。
* 它解决了 **异步同步转换**：将流式的管道通信包装成了类似 HTTP 的“请求-响应”模式。
* 它具备 **环境隔离**：通过之前的 `env -u` 处理，绕过了 EDA 环境中复杂的库冲突问题。

**使用场景举例**：
你可以写一个 Python 脚本循环遍历 100 个 Cell，每次通过这个 TCP 桥接器发指令给 Virtuoso 导出一个 GDS，并实时获取导出是否成功的确认。

### 1. 为什么需要将 stdin 设为非阻塞模式？

在 Linux 系统中，默认读取 `stdin`（标准输入）是**阻塞式**的。这意味着如果管道中没有数据，`sys.stdin.read(1)` 会让整个进程“挂起”，直到收到数据为止。

* **避免“顾此失彼”**：这个守护进程是一个**中转站**。它必须同时监听两个方向：一个是 TCP Socket（来自外部客户端的代码），另一个是 `stdin`（来自 Virtuoso 的返回结果）。
* **并发响应**：如果脚本在等待 Virtuoso 返回结果时被阻塞在 `stdin` 上，它就无法及时处理 TCP 连接请求。
* **轮询机制**：将其设为非阻塞（Non-blocking）后，如果没有数据，读取动作会立即抛出一个 `EAGAIN` 或 `EWOULDBLOCK` 错误。脚本捕获这个错误，稍微休眠一下（`time.sleep(0.001)`），然后继续检查 TCP 或再次尝试读取。这保证了程序在单线程下依然能灵敏地响应双向数据。

---

### 2. 安全过滤器的正则表达式如何工作？

代码中的过滤器逻辑如下：
`_BLOCKED_FNS = re.compile(r'(?<!["\w])(shell|system|ipcBeginProcess|...)\s*\(')`

它的工作机制分为两步：

1.  **预处理（剥离字符串）**：
    `re.sub(r'"[^"]*"', '""', skill)`
    脚本先用正则把所有双引号内的内容替换为空。这是为了防止**误报**。例如，如果你的代码是 `printf("Executing system command")`，这本身是安全的，但如果不剥离字符串，正则会检测到 `system` 这个词。
2.  **负向后瞻断言（Negative Lookbehind）**：
    * `(?<!["\w])`：这部分确保危险关键词（如 `shell`）前面**不能**是双引号或字母数字。这防止了像 `my_shell_var` 这种包含关键词但实际安全的变量名被拦截。
    * `\s*\(`：确保关键词后面跟着括号，这意味着它是一个**函数调用**。

### 3. 如何处理多行 SKILL 代码？

Virtuoso 的 `ipcBeginProcess` 在处理输出流时，默认是**按行执行**回调函数的。如果直接把一段多行的 SKILL 代码扔过去，Virtuoso 可能会将其拆碎成多段执行，导致逻辑错误。

脚本采用了**“压缩平铺”**策略：
* **注释清理**：使用正则 `r';[^\n]*'` 删掉所有分号开头的注释。如果不删掉注释就强行合并行，原本在分号后面的有效代码会被当作注释“吞掉”。
* **换行替换**：将换行符替换为空格。
* **空白压缩**：使用 `skill.split()` 和 `' '.join()` 将冗余的空格压缩。

**结果**：无论你在客户端写得多美观（带缩进、带注释），发给 Virtuoso 的都是一整行精简的指令。这样 Virtuoso 的 `evalstring` 就能一次性吞下整个逻辑块。

### 4. 为什么是单线程设计？优缺点是什么？

这个脚本采用了最简单的“接受一个连接 -> 处理 -> 结束”的单线程阻塞/轮询模式。

**优点：**
* **顺序一致性**：Virtuoso 本身（在很大程度上）是单线程执行 SKILL 的。单线程设计保证了发出的指令和收到的结果是严格一一对应的，不会出现“A 请求的结果被 B 拿走”的竞态条件。
* **低复杂度**：不需要处理多线程的锁（Lock）或资源竞争，降低了在 EDA 这种特殊环境下崩溃的概率。
* **资源占用低**：对于这种中转工具，单线程的 CPU 和内存开销极小。

**缺点：**
* **排队效应**：如果一个复杂的 SKILL 任务需要运行 10 秒（例如导出超大 GDS），在这 10 秒内，其他客户端尝试连接该端口会被挂起或拒绝。
* **吞吐量限制**：无法同时处理多个并发请求。

**总结**：对于 EDA 自动化的场景，**准确性高于并发性**。由于后端 Virtuoso 只有一个 CIW 会话，多线程并行发送指令反而会导致 Virtuoso 内部状态混乱。
